Git & Version Control

NoteBy the end of this section, you should be able to…
  • Locate and interpret the git tab in Positron, including status indicators for staged and unstaged changes
  • Explain the three-step local git workflow: modify → stage → commit
  • Create your first commit using both the Positron GUI and the command line terminal
  • Explain why large data files should be excluded from git repositories and create a .gitignore file to manage this
  • Use core git commands: git status, git add, git commit, git log
  • Explain the difference between local-first and remote-first (clone) repository workflows
  • Generate and add an SSH key to GitHub to authenticate securely
  • Create a remote repository on GitHub.com and push your local commits to it
  • Choose the appropriate GitHub platform (GitHub.com, GitHub Enterprise Cloud, or GitHub Enterprise Server) for a given research scenario
  • Fork a repository, create a branch, propose changes via a Pull Request, and review and merge an incoming Pull Request

Introduction

In this section, we will get hands-on with git right inside Positron. We’ll learn the fundamental workflow of saving changes to our project’s history. This entire process happens locally, on our own computers. We will explore this workflow in two ways: using Positron’s graphical interface and using the command line terminal.

What we’ll cover:

  1. Local git workflow — Making and saving changes to your project history
  2. Best practices — Managing data files with .gitignore to keep repositories lean
  3. Connecting to GitHub — Publishing your local repository to GitHub.com as a remote backup and optionally sharing it publicly
  4. Collaboration — If time permits, we’ll explore how to collaborate with colleagues using Pull Requests (also called Merge Requests), which allow you to propose changes to each other’s projects safely and professionally

1. Recap: The Initialized Repository

When your Positron Project was created with the option to initialize a git repository, git began tracking your project folder immediately.

Your mission control for git in Positron is the Source Control tab, located in the left activity bar (the same panel used in VS Code).

Action: Click on the Source Control tab. If there are no pending changes, the panel will be empty — which means git doesn’t see any unsaved modifications.

Understanding git Status Indicators

When you make changes to files in your project, they appear in the Source Control panel with status indicators. There are two zones:

Staged Changes (What Will Be Committed)

Indicator Meaning What It Tells You
🟢 A Added New file staged for commit
🔵 M Modified Changed file staged for commit
🔴 D Deleted File staged for deletion

Changes (Working Directory — Not Yet Staged)

Indicator Meaning What It Tells You
🟡 M Modified File has unstaged changes
🟡 U Untracked New file git doesn’t know about
🔴 D Deleted File deleted but not staged

Common Combinations

Staged Unstaged What This Means
(empty) U Brand new file, not tracked yet
(empty) M Existing file changed, not staged
A (empty) New file staged, ready to commit
M (empty) Modified file staged, ready to commit
M M Some changes staged, more changes unstaged
TipQuick Reference
  • Staged Changes = What’s packed and ready for your commit
  • Changes = What’s changed but not staged yet
  • Clicking the + icon next to a file moves it from Changes → Staged Changes

2. The git Workflow in Positron

The core git workflow involves three steps:

  1. Make changes to your files
  2. Stage the specific changes you want to save
  3. Commit those staged changes, creating a permanent snapshot in your project’s history

Confirm You Have a git Repository

  1. Check the Source Control tab. Look at the left activity bar in Positron. If you see a Source Control icon with a badge, your repository is initialized. If not, ask your instructor for help.

  2. Open the Terminal. Click on the Terminal tab at the bottom of Positron.

  3. Verify the repository. In the terminal, type:

git status

You should see a message listing “Untracked files” — these are the R scripts and data files you’ve been working with throughout the workshop. This confirms git is watching your project folder but hasn’t saved any snapshots yet.

  1. Check what files you have:
ls

You should see your workshop files like ARCS_Part1.R, ARCS_Part2.R, folders like raw_data/, scripts/, etc.

Your First Commit: Saving Existing Work

  1. Check the Source Control tab. You should see several files listed as untracked (U), indicating git doesn’t know about them yet. You might see files like:
    • ARCS_Part1.R and ARCS_Part1b.R
    • ARCS_Part2.R and ARCS_Part2b.R
    • Shank3_metadata_parsed.csv
    • Various files in your raw_data/ folder
  2. Stage all files. Open the Terminal tab and type:
git add .

The . means “add all changed files in the current directory and subdirectories.”

NoteAbout Line Ending Warnings

You may see warnings like “CRLF will be replaced by LF the next time Git touches it.” This is normal — git is automatically converting Windows-style line endings to Unix-style for consistency across operating systems. You can safely ignore these warnings.

  1. Check the Source Control tab again. All your files should now appear under Staged Changes, meaning they’re ready to commit.

  2. Commit your work. In the Source Control panel, type your commit message in the message box at the top:

    Add workshop scripts and data from Parts 1 and 2

    Then click the ✓ Commit button.

  3. View the history. In the terminal, run:

git log --oneline

You can see your first commit listed!

Congratulations — you’ve created your first snapshot of the project!

Creating New Files for Practice

  1. Create a new file. In Positron, go to File → New File → R Script and save it as ARCS_Part3b.R in your scripts folder.

  2. Add a comment:

# This is a placeholder for our script content

  1. View the change. Go back to the Source Control tab. You’ll see ARCS_Part3b.R listed as untracked (U).

  2. Stage the file. Click the + icon next to ARCS_Part3b.R. It will move to Staged Changes with an A indicator.

  3. Commit the change. Type the message Add initial analysis script in the commit message box and click ✓ Commit.

You’ve completed your second version control cycle!

3. Excluding Data Files: The .gitignore File

Before we continue, let’s discuss an important best practice: not storing large data files in git repositories.

Why Exclude Data Files?

Git is designed to track code and text files, not large data files:

  • Repository bloat: Data files can make your repository huge and slow to clone, push, and pull
  • Version control inefficiency: Git tracks every version of every file — a 100MB dataset becomes 500MB after 5 versions
  • GitHub limitations: GitHub has a 100MB per-file limit and recommends keeping repositories under 1GB
  • Not designed for data: Git compresses text efficiently but struggles with binary data formats
NoteWhat About Data Storage?

For data files, consider these alternatives:

Data Size Recommended Approach
Small reference data (< 10MB) Can include in repository if rarely changes
Medium data (10MB–1GB) Cloud storage (Dropbox, Google Drive, OneDrive) — document the location
Large data (> 1GB) Institutional repositories, OSF, Zenodo, or Figshare
Very large data Git LFS (Large File Storage) — stores large files externally while keeping references in git

Creating a .gitignore File

  1. Create the file. In Positron, go to File → New File → Text File

  2. Add these lines:

# Ignore data folders
raw_data/
processed_data/

# Ignore large output files
outputs/*.pdf
outputs/*.html

# Keep the folder structure but ignore contents
!raw_data/.gitkeep
!processed_data/.gitkeep
!outputs/.gitkeep

  1. Save the file as .gitignore (with the dot!) in your project root folder

  2. Create placeholder files so the folders still exist in git. In the terminal:

touch raw_data/.gitkeep
touch processed_data/.gitkeep
touch outputs/.gitkeep

The .gitkeep files are empty placeholders that preserve your folder structure in the repository even though the data files themselves are ignored.

  1. Commit the .gitignore file. Stage .gitignore and the three .gitkeep files and commit with the message: Add .gitignore to exclude data files
TipVerify What’s Being Tracked

After adding .gitignore, run this to confirm your data files are ignored:

git status --ignored

You should see your data folders listed under “Ignored files”.

ImportantKey Takeaway

Keep your git repository lean by tracking code, not data. Document where your data lives in your README file instead.

4. git on the Command Line — A Quick Look

The Positron Source Control panel is a visual interface that runs git commands for you in the background. Let’s pull back the curtain and run those same commands ourselves.

  1. Open the Terminal in Positron (bottom panel).

  2. Check the project status:

git status

It should say nothing to commit, working tree clean since we just committed all our changes.

  1. Make another change. Go back to ARCS_Part3b.R and add a new line:
# This is our first tracked change.
print("Hello from R!")
  1. Check the status again:
git status

git now sees that ARCS_Part3b.R has been modified.

  1. Stage the change (CLI):
git add .

Run git status once more — you’ll see the file listed under “Changes to be committed.”

  1. Commit the change (CLI):
git commit -m "Add print statement via CLI"

You’ve just done the exact same workflow as before, but entirely from the command line!

NoteKey Takeaway

The Positron Source Control panel is a user-friendly interface for running these fundamental git commands. You can use whichever you prefer, but knowing what’s happening behind the scenes makes you a more confident and capable user.

6. Local-First vs. Remote-First Repositories

Local-First (What We Just Did)

You start a project on your own computer, initialize a git repository, work on your files, and build a commit history — all locally. Later, you can push this project to GitHub to back it up, share it, or collaborate.

Analogy: You write a document in your text editor on your desktop. When you’re ready, you upload it to cloud storage to share with colleagues.

Remote-First (Cloning)

The project already exists on a remote server like GitHub. Your first step is to clone it — downloading a complete copy of the project and its entire version history to your local computer.

Analogy: A colleague shares a document with you via cloud storage. You download a local copy so you can start editing.

Note

In this workshop we started local-first. In the collaboration section below, you will experience the remote-first workflow when you clone a partner’s repository.

7. Publishing Your Repository to GitHub

Now that we have a local repository with a commit history, let’s back it up and share it by publishing to GitHub.com. This requires two steps: setting up SSH authentication and creating a remote repository.

Step 1: Set Up SSH Authentication

SSH (Secure Shell) is a secure way to connect your computer to GitHub. Think of it as a special key that proves your identity without typing your password every time.

NoteAlternative: Personal Access Tokens (PATs)

PATs are an alternative authentication method using HTTPS. However, they expire every 90 days and require renewal, which can disrupt your workflow. SSH is strongly preferred for long-term research projects because it is a one-time setup.

If SSH doesn’t work due to network or firewall restrictions, GitHub Desktop (which uses HTTPS) is a good backup option.

Check for Existing SSH Keys

In the Positron Terminal, run:

ls ~/.ssh

If you see files named id_ed25519 or id_rsa, you already have keys — skip to Add Your Key to GitHub below. If not, continue.

Generate a New SSH Key

  • macOS / Linux
  • Windows
  1. Generate the key (replace with your GitHub email):
ssh-keygen -t ed25519 -C "your_email@example.com"

  1. Save the key. Press Enter to accept the default location.

  2. Set a passphrase (optional). Press Enter twice to skip, or type a passphrase you’ll remember.

  3. Start the SSH agent:

eval "$(ssh-agent -s)"
  1. Add your key:
ssh-add ~/.ssh/id_ed25519
  1. Copy your public key:
pbcopy < ~/.ssh/id_ed25519.pub
NoteWindows Users: Use the Positron Terminal

These commands work in the Positron Terminal or git Bash (installed with git for Windows). We recommend using the Positron Terminal for consistency.

  1. Generate the key (replace with your GitHub email):
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"

  1. Save the key. Press Enter to accept the default location.

  2. Set a passphrase (optional). Press Enter twice to skip.

  3. Start the SSH agent:

eval "$(ssh-agent -s)"
  1. Add your key:
ssh-add ~/.ssh/id_rsa
  1. Copy your public key:
cat ~/.ssh/id_rsa.pub | clip

If clip doesn’t work, run cat ~/.ssh/id_rsa.pub and manually copy the output.

WarningWindows Troubleshooting

If ssh-agent won’t start, try running Positron or git Bash as Administrator. If you see “Could not open a connection to your authentication agent,” run ssh-agent bash first, then retry ssh-add.

ImportantKeep Your Keys Safe!
  • The .pub file is your public key — safe to share with GitHub
  • The file without .pub is your private key — never share this
  • If someone asks for your SSH key, they always mean the public one

Add Your SSH Key to GitHub

  1. Go to github.com and log in
  2. Click your profile photo → Settings
  3. In the left sidebar, click SSH and GPG keys
  4. Click New SSH key
  5. Add a descriptive title like My Positron Laptop
  6. Paste your key in the Key field
  7. Click Add SSH key

Test Your Connection

ssh -T git@github.com

Type yes if prompted about authenticity. You should see: Hi username! You've successfully authenticated...

Step 2: Create a Repository on GitHub.com

  1. Go to github.com and log in
  2. Click your profile picture → Your repositories
  3. Click the green New button
  4. Repository name: e.g., ARCS-workshop
  5. Description (optional): Reproducibility workshop project
  6. Visibility: Private (or Public if you want others to see it)
  7. Important: Do NOT check “Add a README file” — we already have files locally
  8. Click Create repository

Step 3: Connect and Push

GitHub will show setup instructions for “push an existing repository from the command line.” Run these one at a time in your Positron Terminal (replace username and yourreponame with yours):

git remote add origin git@github.com:username/yourreponame.git
git branch -M main
git push -u origin main

Refresh your GitHub repository page — you should see all your files and commit history.

TipTroubleshooting: GitHub Desktop as a Backup Method

If you encounter authorization issues, GitHub Desktop is a reliable alternative:

  1. Download and install GitHub Desktop
  2. Sign in with your GitHub account
  3. Click Add → Add Existing Repository and navigate to your project folder
  4. Click Publish repository

This method uses HTTPS authentication instead of SSH.

8. Bonus: Collaboration with GitHub Desktop & Positron

Now let’s explore one of the most powerful aspects of version control: collaboration. This section demonstrates a complete, real-world collaborative workflow where one person proposes a change to another person’s project.

Understanding the Roles

  • Project Owner: Owns the original repository and has authority to accept or reject changes
  • Contributor: Wants to suggest improvements, fixes, or additions

The Fork-and-Pull Request Workflow

  1. Fork — Contributor creates their own complete copy of the project
  2. Clone — They download their copy to work on locally
  3. Branch — They create a separate branch for their contribution
  4. Edit — They make changes and commit them
  5. Push — They upload changes to their fork on GitHub
  6. Pull Request — They formally ask the owner to review and merge

Part 1: The Contributor’s Workflow

Step 1: Fork the Repository

  1. Navigate to the project owner’s repository on GitHub.com
  2. Click the Fork button in the top-right corner
  3. GitHub creates a copy of the entire project under your own account
NoteWhat Forking Actually Does

Forking creates an exact copy of the entire project history — all commits, branches, and files — under your account. You can experiment freely without affecting the original project.

Step 2: Clone Your Fork to GitHub Desktop

  1. In your forked repository, click the green <> Code button
  2. Copy the HTTPS URL
  3. Open GitHub Desktop → File → Clone Repository
  4. Paste the URL, choose a save location, and click Clone

Step 3: Create a New Branch

  1. In GitHub Desktop, click the Current Branch dropdown
  2. Click New Branch
  3. Give it a descriptive name (e.g., add-readme-documentation)
  4. Click Create Branch
  5. Click Publish branch
TipBranch Naming Conventions

Good branch names are descriptive, verb-focused, and lowercase with hyphens:

  • ✓ add-ggplot-visualization
  • ✓ fix-data-loading
  • ✗ new-feature
  • ✗ MyBranch

Step 4: Make Your Edits in Positron

  1. In GitHub Desktop, click Open in Positron (or open the .Rproj file directly)

  2. Create a new Markdown file: File → New File → Markdown File

  3. Save it as README.md in the project root folder

  4. Add the following content:

# Introduction to Programming in R: A Workshop Series 💻

"Introduction to Programming in R" workshop series, part of the
**Advancing Reproducibility with Computational Skills (ARCS)**

NoteWhy Add a README?

A README is the first thing people see when they visit a repository. It should explain what the project is about, how to get started, what prerequisites are needed, and how the project is organized. Good documentation makes your research more accessible and reproducible.

Step 5: Commit and Push

  1. Switch back to GitHub Desktop
  2. Review your changes by clicking the file names in the left panel
  3. Write a commit message: Add README documentation for workshop series
  4. Click Commit to [your-branch-name]
  5. Click Push origin

Step 6: Open a Pull Request

  1. After pushing, click Create Pull Request in GitHub Desktop
  2. Give your PR a clear, descriptive title
  3. In the description, explain what you changed and why
  4. Click Create pull request

Example PR description:

## Summary
Added comprehensive README documentation to help users understand
the ARCS workshop series structure, objectives, and prerequisites.

## Changes Made
- Created README.md in the project root
- Documented all three workshop parts with learning objectives
- Listed prerequisites and installation links
- Updated RStudio references to Positron throughout

## Why This Improvement
The repository lacked documentation explaining what the series
covers, how the three parts build on each other, and what software
participants need.

Part 2: The Project Owner’s Review Process

Step 7: Review and Manage the Pull Request

  1. Navigate to your original repository on GitHub.com
  2. Click the Pull requests tab
  3. Click the specific PR to review it
  4. In Files changed, review the exact differences:
    • Green lines = additions
    • Red lines = deletions
    • Leave line-by-line comments for discussion
  5. In Conversation, ask questions, request changes, or approve

Review options:

  • Approve and Merge: Click Merge pull request → Confirm merge
  • Request Changes: Leave comments asking for modifications first
  • Close without Merging: If the contribution doesn’t fit the project’s direction
ImportantCode Review Best Practices
  • Be constructive: Focus on the code, not the person
  • Be specific: Point to exact lines and suggest improvements
  • Be appreciative: Thank contributors for their time and effort
  • Test locally: For important changes, pull the branch and test it yourself before merging

After Merging

For the Project Owner: - The contribution is now part of the main project - The contributor’s branch can be safely deleted

For the Contributor: - Switch back to main and pull the latest changes - Delete your feature branch — it’s no longer needed - Your contribution is now part of the official project history!

Key Benefits of This Workflow

  • Safety: The original project is never at risk during development
  • Review: All changes go through a review process before integration
  • Discussion: Contributors and maintainers can discuss improvements openly
  • History: Every contribution is documented with clear attribution
  • Learning: Contributors improve their skills through feedback
