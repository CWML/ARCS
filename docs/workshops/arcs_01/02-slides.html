Practical implications

Excel is looping behind a friendly interface, which means that:

Excel hits walls at:

What Are R Projects?

  • Self-contained workspace for each analysis or research project

  • Automatic working directory management. No more setwd()

  • Seamless integration with version control and documentation tools

Think of an R Project like having a dedicated laboratory bench for each experiment. Just like you wouldn’t mix chemicals from different experiments on the same bench, you shouldn’t mix files from different analyses in the same folder. When you open an R Project, R automatically “knows” where all your files are located - your data, scripts, outputs, and documentation all live together. This eliminates one of the most common beginner frustrations: the dreaded “Error: cannot open file” because R can’t find your data. The working directory is automatically set to your project folder, so file paths just work.

WHEN and WHERE to Create R Projects

  • WHEN: At the very beginning, before writing any code

  • WHERE: Create a dedicated folder for each project inside of one “Code” or “R_Projects” folder

  • One R project = one research question/paper/analysis (not one project for all your R work)

The biggest mistake beginners make is creating one giant “R_Work” folder and throwing everything in there. Instead, think of each research question or manuscript as deserving its own project. Create the R Project FIRST, before you even download your data. This is like setting up your lab notebook before starting an experiment - it ensures everything stays organized from day one. Choose descriptive names that will make sense to you (and your collaborators) months later. A good rule of thumb: if you might publish this as a separate paper, it should be its own R Project. The folder location should be somewhere logical on your computer - many people create a “Research” or “Projects” folder in their Documents.

Data Management Advantages to R Projects

  • Relative file paths work everywhere

    • input_file <- "raw_data/GSE113754_GeneLevel_Raw_data.xlsx" #this

    • input_file <- "C:/Users/Username/Code/ARCS/raw_data/GSE113754_GeneLevel_Raw_data.xlsx" #NOT this

  • Share projects easily (zip the folder, send to collaborators/your future self)

  • Clear project boundaries (everything stays together)

Here’s where R Projects really shine for reproducibility. When you use relative paths like “data/metadata.csv”, your code works on any computer - yours, your advisor’s, your collaborator’s - as long as they have the project folder. No more changing file paths when you switch from your laptop to the lab computer! The automatic working directory means you’ll never again see that frustrated student debugging session where half the time is spent figuring out where R is looking for files. When you want to share your analysis, just zip the entire project folder - everything your collaborator needs is in one place. RStudio keeps track of your recent projects, so you can easily switch between your Shank3 analysis and your methods comparison without confusion. This organization becomes crucial when you’re juggling multiple research projects simultaneously.

What is Git?

  • Version control system - like track changes for your entire project, not just one document

  • Takes snapshots of your project at different points in time

  • Remembers what changed

  • Works locally on your computer

  • Different from GitHub - Git is the technology, GitHub is the website

  • Every R Project can use Git

The .gitignore file is our first encounter with version control using git. Has anyone heard about this technology or used it before? What do you know about it? This is a technology that you will learn more about through the GitHub platform in part 3 of this class.

For now, you just need to know that git is a more powerful way of doing something like tracking changes and resolving differences if you and perhaps your collaborators iterate through different versions of a script and expand your project, or, if you need to roll back to an earlier version of your project, you can do that with Git. Git creates an audit-proof version of record for the coding work that you do. It can be one very important part of making your work reproducible.

What we just did in this exercise is build the foundation of reproducible computational research. We created two things that work together: structure and documentation.

The file structure- separating raw data from processed data, organizing scripts and outputs - that’s not just about being neat. It creates a clear audit trail that anyone can follow. Your raw data folder is like evidence that stays untouched, while your folder structure shows exactly how data flows through your analysis.

The README we created is what makes your structured project actually usable. You could have the most organized folders in the world, but without documentation explaining what everything does and how to use it, it’s still a mystery to others - and to your future self.

Together, these create true computational reproducibility. Someone else can look at your project, quickly understand what you did, run your analysis, and get the same results. That’s the gold standard in research - work that others can independently verify.

Six months from now, when you’ve forgotten the details, this combination of clear organization and good documentation will be your lifeline. Ideally, you’ll open your project, read your README, and immediately know exactly what you were doing and how to continue.

This isn’t just good practice - it’s what separates amateur data analysis from trustworthy research, and what protects you as a researcher when funders, publishers, and peers ask you to show your work.

File Naming Conventions

  • No spaces (use underscore or en dash)

  • Start with dates using International Organization for Standardization (ISO)

    • YYYYMMDDHHMMSS

  • Use descriptive names

Let’s talk about something that seems simple but will save you hours of frustration: naming your files well.

First rule: no spaces in file names. Spaces are often special characters in programming languages that have a specific assigned use behind the scenes. Or, they may keep your computer from understanding that words belong together. Use underscores or dashes instead - both work fine, just pick one and stick with it.

For dates, I want you to learn the ISO standard: YYYY-MM-DD. So October 2, 2025 becomes 20251002. Why? Because when your computer sorts files alphabetically, they’ll automatically sort in chronological order too. The American date format won’t do this.

Be descriptive with your names. In six months when you’re looking through your files, ‘data.csv’ tells you nothing, but ‘participant_demographics.csv’ immediately tells you what’s inside.

The key is consistency. Whatever naming pattern you choose, use it for everything in your project. Your future self will thank you when you can find files instantly instead of opening each one to see what’s inside.”

File Naming Conventions

❌ Bad Examples:

  • data analysis.R (spaces)

  • data.csv (not descriptive)

  • final_FINAL_really_final.R (version chaos)

✅ Good Examples:

  • 2024-03-15_survey_analysis.R

  • participant_demographics_clean.csv

  • README.md
